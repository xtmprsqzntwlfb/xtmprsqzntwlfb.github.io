The Elephant in the Room: Why Horizon Still Runs on AngularJS

August 15, 2024

On December 31, 2021, Google officially ended support for AngularJS. No more security patches. No more bug fixes. No more anything. The framework was declared dead.

As of 2024, OpenStack Horizon — the dashboard used by thousands of OpenStack deployments worldwide — is still built on it.

Let me explain why that's not as crazy as it sounds.

The Scale of the Problem

This isn't a few components sprinkled in for interactivity. AngularJS is deeply woven into Horizon's architecture. Entire panels are Angular-based:

  • The Images panel
  • The Launch Instance workflow (with its multi-step wizard)
  • Network topology views
  • Countless modal dialogs and schema forms

The code is organized in three tiers: dashboard-specific code, application-shared utilities, and framework-reusable components. Angular module names map directly to the directory structure. It's not a mess — it's actually well-organized, following the John Papa style guide, with proper testing and linting.

The problem is there's just so much of it.

Why "Just Upgrade to Angular 2" Doesn't Work

Here's the thing that catches people off guard: Angular 2+ is not AngularJS 2.0. It's a completely different framework that happens to share a name.

Between Angular 1.x (AngularJS) and Angular 2, Google rewrote everything from scratch. Different architecture (components vs. controllers), different syntax (TypeScript vs. JavaScript), different tooling (CLI vs. Bower), different everything.

"Upgrading" from AngularJS to modern Angular is the same amount of work as migrating to React or Vue. You're rewriting your frontend.

There is an incremental migration path called ngUpgrade that lets you run both frameworks side-by-side, gradually porting components. It's meant for large applications that can't afford a big-bang rewrite. But here's the catch: you still have to rewrite every component eventually. It just spreads the pain over months or years instead of weeks.

The Plugin Ecosystem

Horizon isn't just the core dashboard. It's a platform. There are dozens of plugins for different OpenStack services:

  • Octavia Dashboard for load balancers
  • Manila UI for shared filesystems
  • Magnum UI for container orchestration
  • And many more

These plugins depend on Horizon's framework code. When you write a Horizon plugin, you use the same AngularJS patterns, the same widgets, the same schema forms. It's part of the API.

Migrating Horizon to a new framework doesn't just mean rewriting Horizon. It means either:

  • Breaking every plugin (and asking each maintainer to rewrite)
  • Supporting two frameworks in parallel (the ngUpgrade approach, but across an entire ecosystem)
  • Building a compatibility layer (which defeats the purpose of modernizing)

None of these options are appealing.

The Cost-Benefit Analysis

Let's be honest about what we'd gain from a migration:

Benefits:

  • Modern tooling and developer experience
  • Easier to recruit developers (nobody wants to learn AngularJS in 2024)
  • Better performance (maybe — modern frameworks are faster, but it depends on what you're doing)
  • Active community support and ecosystem

Costs:

  • Months of development time to rewrite the frontend
  • Testing everything to ensure feature parity
  • Plugin ecosystem disruption
  • Risk of introducing new bugs in stable, working code
  • Opportunity cost — that time could be spent on new features

The hard truth? For a project like Horizon, the costs are enormous and the benefits are mostly about developer convenience, not user-facing improvements.

The Stability Paradox

Here's the weird thing about running on dead technology: it's actually quite stable.

AngularJS 1.8.2, the final version, was released in 2020. We migrated to it in early 2023, cleaning up the API changes and deprecated patterns along the way. Since then? It just works.

No new versions to track. No breaking changes to adapt to. No framework churn. The code we write today will work the same way next year, because the framework literally cannot change — it's frozen.

There's something liberating about that. We're not on the upgrade treadmill anymore.

Of course, there's a flip side. No new features. No performance improvements. No security fixes from upstream. We're on our own.

Living in Maintenance Mode

So what's the strategy? We're in what I'd call "active maintenance mode":

Security: We stay on the latest (and final) AngularJS version, 1.8.2, which includes all known security fixes. If a new vulnerability is discovered, we'd have to patch it ourselves — but AngularJS is mature and battle-tested code. The attack surface is well-understood.

Development: We're not expanding the AngularJS footprint. New features go into Django views when possible. When we must write JavaScript, we keep it simple and avoid creating new Angular dependencies.

Recruitment: This is the hardest part. As one mailing list discussion put it in 2020: "it's very hard to get anyone enthusiastic about working on Horizon" because of the AngularJS codebase. That hasn't changed. We frame it honestly — you'll be maintaining legacy code, but you'll also be keeping critical infrastructure running for thousands of deployments.

The Long Game: Eventually, we'll need to migrate. But it'll happen when:

  • The pain of staying exceeds the pain of moving
  • We have the resources for a multi-month rewrite effort
  • We can coordinate with the plugin ecosystem
  • There's a clear target framework

Until then, we're doing something that sounds absurd but actually works: running production systems on a dead framework.

The Broader Lesson

The tech industry loves to talk about modernization, about staying current, about avoiding technical debt. And those are real concerns — technical debt compounds, old frameworks become security liabilities, recruitment gets harder.

But there's a counter-narrative that doesn't get enough airtime: sometimes the old thing just works. Sometimes the stability of a frozen, well-understood codebase is worth more than the shiny new framework. Sometimes "good enough and stable" beats "modern but churning."

Horizon on AngularJS is technical debt by any reasonable definition. But it's technical debt we've chosen to carry because the alternative — a multi-month rewrite with ecosystem-wide coordination — is more expensive than the interest payments.

We'll pay it off eventually. Just not today.

What About You?

If you're running AngularJS in production in 2024, you're not alone. Plenty of applications are in the same boat — internal tools, enterprise dashboards, embedded UIs in larger systems.

The question isn't "should we have migrated already?" It's "what's the right time for us?" And the answer depends on:

  • How much AngularJS code you have
  • Whether it's still meeting user needs
  • How much runway you have before a real crisis (security vulnerability, browser compatibility break)
  • What else you could be building with that migration time

For Horizon, the math says: keep running, keep monitoring, and plan for an eventual migration when the conditions are right.

In the meantime, we'll be here, quietly maintaining the elephant in the room.
