When Good Dependencies Go Stale: Internalizing i18n Extraction in OpenStack Horizon

April 22, 2026

Sometimes the best way to fix a dependency problem is to delete the dependency entirely. This is the story of how OpenStack Horizon moved from enmerkar to internal Django template message extraction, implementing the functionality in 181 lines of code.

The Problem: When setuptools Moves On

It started innocently enough. Setuptools 82.0.0 was released, and suddenly our CI was red. The culprit? The pkg_resources module had been removed from setuptools after years of deprecation warnings. Most projects had already migrated to importlib.resources and importlib.metadata, but we had a problem: enmerkar, our Django template message extractor, still depended on pkg_resources.

The quick fix was obvious: 

# tox.ini
setuptools<82  # Required by enmerkar

But pinning setuptools felt wrong. This wasn't a "we need time to migrate" pin — this was a "we're waiting for an external dependency to catch up" pin. And looking at enmerkar's repository, the outlook wasn't great.

Understanding Enmerkar

For context, enmerkar is a fork of django-babel, which itself had gone quiet years ago with pull requests for Django compatibility piling up unanswered. When Horizon migrated to enmerkar around version 18.0.0, it was the right choice — a maintained fork that worked with modern Django.

Enmerkar's job is straightforward: extract translatable messages from Django templates. When you run tox -e manage -- extract_messages, it scans your templates for patterns like: 

{% trans "Click here to continue" %}

{% blocktrans %}
  Welcome, {{ name }}!
{% endblocktrans %}

{{ _("Server created") }}

It finds these strings and dumps them into a Portable Object Template (POT) file that translators can work with. Critical infrastructure, runs on every build, touches the entire codebase.

The "Wait, How Much Code Is This?" Moment

The issue was brought to the openstack-discuss mailing list after the propose-translation-update job started failing. A fix PR had been sitting in enmerkar's repository for over a month with no response from maintainers.

Several options were discussed, from quick hot-fixes (hardcoding setuptools<82 in the job script, applying the importlib.metadata fix during CI) to more fundamental solutions like forking enmerkar. One option stood out: copy the extraction logic directly into Horizon.

We decided to actually look at what enmerkar was doing for us. How much complexity were we depending on?

Turns out: not much. The core extraction logic was a single function around 200 lines, with no dependencies beyond Django itself. It parsed Django template tokens, looked for translation tags, and yielded them to Babel. It's BSD 3-Clause licensed, so copying it was legally straightforward. And critically, registering it as a Babel extractor entry point meant Horizon plugins would automatically benefit without needing to update their own requirements.

There was an important constraint: most Horizon plugins don't declare enmerkar in their own requirements.txt — they depend on Horizon's requirements. Any solution had to work for the entire ecosystem, not just the core project.

The decision became clear: why wait for an external project to fix setuptools compatibility when we could internalize this critical piece of infrastructure in an afternoon?

The Implementation

The new extraction module lives in horizon/utils/babel_extract_django.py. Here's the core function signature: 

def extract_django(fileobj, keywords, comment_tags, options):
    """Extract messages from Django template files.

    :param fileobj: the file-like object the messages should be
                    extracted from
    :param keywords: a list of keywords (i.e. function names) that
                     should be recognized as translation functions
    :param comment_tags: a list of translator tags to search for and
                         include in the results
    :param options: a dictionary of additional options (optional)
    :return: an iterator over (lineno, funcname, message, comments)
             tuples
    """

The function handles Django's template tokens — TEXT, VAR, and BLOCK types — and recognizes several translation patterns:

  • {% trans %} — inline translations
  • {% blocktrans %} — block-level translations with variable interpolation
  • Context-aware translations with pgettext and npgettext
  • Plural forms with ngettext

The trickiest part? Handling blocktrans with variables: 

{% blocktrans with name=user.name %}
  Welcome, {{ name }}!
{% endblocktrans %}

This needs to become "Welcome, %(name)s!" in the POT file so translators know where the variable goes. The extraction logic tracks state across tokens, building up the message string and variable mappings as it walks through the block.

Plugging Into Babel

Babel supports custom extractors via entry points in pyproject.toml: 

[project.entry-points."babel.extractors"]
django = "horizon.utils.babel_extract_django:extract_django"

This is the same mechanism enmerkar used. When you run pybabel extract, Babel sees this entry point and uses our function for files matching the Django template pattern. No changes needed to babel-django.cfg or our extraction workflow.

The Cleanup

With the extraction logic internalized, we could finally delete: 

# requirements.txt
- enmerkar>=0.7.1  # BSD

# tox.ini
- setuptools<82  # Required by enmerkar

Two dependencies gone. One constraint removed. Zero regressions.

Lessons Learned

The Threshold

Not every dependency needs to be a dependency. When you're pinning other packages to work around a third-party issue, it's time to ask: "How much code are we actually using here?" Sometimes reimplementation is cheaper than the coordination cost of waiting for upstream fixes.

Critical Paths Should Be Internal

Message extraction runs on every build. It touches every template. It's in the hot path for releases. Infrastructure this critical deserves to be under your direct control, not hostage to the maintenance bandwidth of an external project.

Dependency Hygiene

Dependencies are like servers: they require maintenance, they accumulate cruft, and eventually they fail. The difference is you can't just apt-get upgrade an abandoned Python package. Regular dependency audits aren't just about security — they're about sustainability.

The Setuptools Evolution

The removal of pkg_resources is part of a broader modernization in Python packaging. Projects still using pkg_resources need to migrate to:

  • importlib.metadata — for package metadata
  • importlib.resources — for accessing package data
  • packaging — for version parsing and specifiers

The setuptools maintainers gave years of deprecation warnings, but the ecosystem still had stragglers. When a foundational tool moves on, you either move with it or get left behind.

Was It Worth It?

Absolutely. We traded:

  • An external dependency with uncertain maintenance —> 181 lines of code we control
  • A setuptools version pin blocking updates —> Freedom to upgrade
  • Hoping upstream fixes compatibility —> Fixing it ourselves in one afternoon

The new code is tested, documented, and lives right next to the code that uses it. When Django's template syntax changes, we'll see it in our tests. When Babel updates its extractor API, we'll adapt immediately.

Sometimes the best architecture is the one with fewer moving parts.
